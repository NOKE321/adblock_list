## 数仓项目

### 项目介绍

```sql
# 朴朴超市数据仓库
1、朴朴超市是国内大型即时生鲜电商平台，业务覆盖生鲜食品、日用百货等多个品类，涵盖了从用户下单、仓储拣货到骑手配送的完整业务链路。
2、随着公司业务规模的高速增长，出现了几个核心痛点：
 2-1、第一，数据孤岛问题严重。订单、商品中心、仓储及配送系统数据分散，导致跨业务线的指标口径不统一。
 2-2、第二，业务分析效率低下，取数成本极高，缺乏一个统一的数据底座来支撑经营决策。
3、3、我们基于 Kimball 维度建模方法论，搭建了面向商品经营分析的离线数仓，沉淀交易、履约、商品、库存、营销主题域五大主题域，并引入 presto给数分同事作为即席查询引擎。
```



### 技术栈与数据流转

```sql
# 朴朴超市数据仓库
# 技术栈
1、存储层：用 HDFS 做底层存储，Hive 作为数仓主体，元数据走 Hive Metastore
2、计算引擎：Hive 默认的 MR 引擎太慢，我们用 Spark 替换了 MR，所有 Hive SQL 通过 Spark 引擎
执行，TB 级任务执行效率显著提升
3、资源调度：YARN 做资源管理，划分了不同业务线的资源队列，避免大任务抢占
4、任务调度：用 DolphinScheduler 做工作流编排，管理 150+ 离线任务的依赖、重试、告警
5、即席查询：分析师高频取数场景用presto，秒级响应
6、BI 展示：业务报表和大屏用 Finebi


# 数据流转
数据源主要分两类：业务数据存在各业务系统的 MySQL 里，比如订单、支付、退款、商品、库存、履约、营销、用户等；日志数据主要是 App 埋点日志、用户行为日志、搜索曝光点击日志等存在日志服务器或通过埋点服务统一采集。

1、业务数据走 DataX 入仓
业务数据是结构化的，T+1 同步基本够用。我们根据表特点做区分：商品、品类、前置仓、城市、活动配置这种维度小表，每天凌晨做全量同步；订单、支付流水、退款售后、库存流水、履约节点这种大表，基于update_time 做增量抽取，按天分区落到 ODS。整个同步通过调度平台凌晨调度，避开业务高峰，减少对线上业务库的影响。

2、日志数据走 Flume + Kafka 入仓
日志量大、实时性要求高，所以用流式采集。Flume 部署在日志服务器实时采集日志，写到 Kafka 做缓冲削峰，再消费 Kafka 把数据落到 HDFS，挂到 Hive 外部表。用 Kafka 做中间层主要是为了应对活动期间日志量突增，也方便后续接入实时链路。

数据落到 ODS 之后，Hive + Spark 做离线加工， 数据开发平台调度全链路任务和进行DQC数据质量监控，最终结果一部分给 presto做即席查询，一部分对接 Finebi 出报表。

```



### 数仓建设

```sql
# 数仓分层
ODS：数据源层，保留原始数据，不做加工
DIM：数据维度层，沉淀一致性维度
DWD：数据明细层，基于业务过程构建明细事实表
DWS：数据汇总层，按主题做轻度汇总，沉淀公共dw指标
ADS：数据应用层，面向具体报表需求做最后加工
```







## 画像项目

### 项目介绍

```sql
# 朴朴超市用户画像平台
1、即时生鲜电商属于高频快消领域，用户群体对“满减促销”、“配送券”等营销手段极为敏感。
2、在建设画像平台之前，业务侧在营销上存在两个明显的痛点：
 2-1、第一，营销手段粗放。过去运营团队往往采用盲目群发短信、全员派发大额券的方式进行促活，没有针对性的触达策略。
 2-2、第二，营销成本高但转化率低（ROI低下）。因为缺乏系统化的用户特征数据支撑，无法精准识别不同偏好的用户群体，导致营销资源浪费。
3、为了解决这些问题，我们团队建设了支撑精细化运营的用户画像平台。通过清洗数仓的交易与流量日志，从人口属性、行为特征等维度沉淀了多个高价值标签；同时，针对营销中台复杂的多维人群圈选场景，我们引入了 Bitmap 技术对底层宽表进行优化，将海量用户的圈选查询时效大幅优化到了秒级响应，最终全面支撑了业务线的精准 Push 和千人千面发券。
```

### 技术栈与数据流转

```sql
# 朴朴超市用户画像平台
# 技术栈
1、存储层：底层用 HDFS + Hive 存储用户基础数据、交易明细、行为日志、标签明细表和画像宽表，Hive 作为离线标签加工的主体存储。

2、计算引擎：离线标签主要通过 Hive SQL / Spark SQL 计算。像用户基础标签、交易偏好标签、品类偏好标签、RFM 标签、生命周期标签、优惠券敏感标签等，都是基于 Spark 对历史交易和行为数据进行聚合加工。

3、标签结果存储：离线计算后的标签结果一部分沉淀到 Hive 标签表和画像宽表中，用于后续离线分析；一部分同步到 HBase，用于根据 user_id 快速查询单个用户的标签画像。

4、人群圈选：针对营销中台多标签组合圈选的场景，将常用标签结果同步到 ClickHouse，并结合 Bitmap 思路对用户集合进行预聚合。圈选时通过标签条件的交集、并集、差集快速计算目标用户人群，提升多维标签筛选效率。

5、任务调度：通过 DolphinScheduler 调度每日标签加工任务，管理标签任务之间的依赖、重试、告警和补数，保证画像标签 T+1 稳定产出。

6、数据质量：通过数据质量规则监控核心标签任务，从完整性、准确性、及时性等维度校验标签结果，并配置多级告警，保障营销触达和推荐场景使用的数据稳定可靠。


# 数据流转
用户画像平台的数据源主要分三类：用户基础数据来自用户系统 MySQL，交易数据来自订单、支付、退款、优惠券、营销活动等业务系统，行为数据来自 App 埋点日志、搜索曝光点击日志、加购日志、页面访问日志和营销触达日志。

1、基础数据和交易数据通过 DataX 从 MySQL 同步到 ODS。用户、会员、城市、前置仓等维度表每天全量同步；订单、支付、退款、优惠券核销等事实表基于 update_time 做增量抽取，按天分区落到 ODS。

2、行为日志通过 Flume + Kafka 采集入仓。Flume 从日志服务器实时采集 App 行为日志，写入 Kafka 做缓冲削峰，再消费 Kafka 落到 HDFS，挂 Hive 外部表。

3、数据进入 ODS 后，通过 Hive + Spark 清洗到 DWD 层，形成订单明细、支付明细、优惠券使用明细、用户行为明细、商品曝光点击明细等表，同时做 user_id、device_id、会员 ID 的统一映射。

4、在 DWS 层按用户粒度沉淀用户主题宽表，比如近 7 天访问次数、近 30 天下单次数、近 30 天 GMV、最近一次下单时间、常购品类、优惠券使用次数、活动参与次数等。

5、标签层基于用户主题宽表计算用户标签，包括基础属性标签、行为偏好标签、交易价值标签、RFM 标签、生命周期标签、优惠券敏感标签、品类偏好标签等。标签采用“竖表 + 横表”结合的模式存储，竖表方便标签扩展和管理，横表方便下游查询使用。

6、标签结果一部分写入 HBase，用于根据 user_id 查询单个用户画像；另一部分写入 ClickHouse，并通过 Bitmap 技术支撑营销中台多标签组合圈选，实现人群包秒级返回。

7、最终人群包同步给营销中台，用于 Push、短信、优惠券发放和活动触达；标签数据也提供给运营和数据分析同事做用户分群、活动复盘、推荐策略和流失召回分析。
```
